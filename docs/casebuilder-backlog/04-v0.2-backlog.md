# 04 - V0.2 Backlog

V0.2 focuses on richer WorkProduct profiles: motions, declarations, exhibit lists, export packets, and stronger authority retrieval.

## CB-V02-001 - Motion profile
- Priority: P1
- Area: Motions
- Problem: Motion workflows are not implemented.
- Expected behavior: Build `WorkProduct(type="motion")` with motion outline, legal standard, fact-to-standard map, argument, declaration support, exhibits, proposed order, citation check, and QC.
- Implementation notes: Support generic motion first as a WorkProduct profile, then specialized templates/rule packs.
- Acceptance checks: User can create a generic motion WorkProduct from selected claims/facts/evidence/authority.
- Dependencies: WorkProduct Builder, authority search, evidence matrix, fact checks.
- Status: Partial
- Progress: Generic `WorkProduct(type="motion")` creation returns a canonical AST and is covered by the AST smoke path. Motion-specific frontend profile, templates, support inspector, rule pack, and export polish remain.

## CB-V02-002 - Motion templates
- Priority: P2
- Area: Motions
- Problem: Different motion types need different required sections and standards.
- Expected behavior: WorkProduct templates for motion to dismiss, summary judgment, compel, TRO/preliminary injunction, amend, default, sanctions, and generic motion.
- Implementation notes: Keep templates jurisdiction-neutral unless source rules are available.
- Acceptance checks: Template creates expected WorkProduct outline and required support checklist.
- Dependencies: `CB-V02-001`.
- Status: Todo

## CB-V02-003 - Declaration and affidavit profiles
- Priority: P1
- Area: Declarations
- Problem: Users need witness statements tied to facts/evidence.
- Expected behavior: Build `WorkProduct(type="declaration")` or `WorkProduct(type="affidavit")` with declarant identity, numbered factual statements, exhibit references, verification/signature block, and support checks.
- Implementation notes: Reuse WorkProduct, Evidence, SourceSpan, and shared support chips with declaration-specific profile controls.
- Acceptance checks: Declaration WorkProduct can be generated from selected facts and checked for support.
- Dependencies: V0 facts, evidence, WorkProduct Builder.
- Status: Todo

## CB-V02-004 - Exhibit list profile
- Priority: P1
- Area: Exhibits
- Problem: Exhibit labels exist in data but no full exhibit workflow exists.
- Expected behavior: Assign exhibit numbers, generate `WorkProduct(type="exhibit_list")`, link exhibits to facts/WorkProduct blocks, and build export packet order.
- Implementation notes: Add Exhibit node if CaseDocument `is_exhibit` becomes insufficient; preserve stable exhibit labels when packet order changes. Complaint paragraph/exhibit references are tracked in `CB-CE-008`, shared chip behavior in `CB-WPB-010`, and filing packet use in `CB-CE-018`.
- Acceptance checks: Exhibit list matches document labels and WorkProduct references.
- Dependencies: V0 documents, evidence, WorkProduct Builder, `CB-CE-008`, `CB-CE-018`.
- Status: Todo

## CB-V02-005 - Filing packet profile
- Priority: P1
- Area: Export
- Problem: Export routes are deferred stubs.
- Expected behavior: Build packet containing primary WorkProduct, exhibits, exhibit list, declarations, proposed order, and QC summary.
- Implementation notes: Do not implement court e-filing yet; expose packet preview, status, warnings, and downloadable generated artifacts. Complaint-specific packet assembly is tracked in `CB-CE-018`.
- Acceptance checks: Packet preview shows included files, order, warnings, and missing items.
- Dependencies: `CB-V02-003`, `CB-V02-004`, `CB-CE-018`.
- Status: Todo

## CB-V02-006 - DOCX export
- Priority: P1
- Area: Export
- Problem: Legal work product needs editable export, and the current DOCX path is only a review-needed skeleton.
- Expected behavior: Export WorkProducts and exhibit lists to DOCX with basic pleading-friendly structure.
- Implementation notes: Use a document generation pipeline with visual/render verification. All DOCX generation must consume the shared WorkProduct AST/profile layout path from `CB-WPB-012`; complaint-specific layout work remains referenced by `CB-CE-013` through `CB-CE-015`.
- Acceptance checks: Generated DOCX opens and preserves headings, paragraphs, numbering, citations, and exhibit references.
- Dependencies: Stable WorkProduct model, `CB-WPB-012`, `CB-WPB-040`, `CB-WPB-042`, `CB-CE-013`, `CB-CE-015`.
- Status: Partial
- Progress: WorkProduct export artifacts can represent DOCX skeletons with immutable snapshot/hash metadata and warnings. Production DOCX rendering and visual verification remain.

## CB-V02-007 - PDF export

## CB-V02-008 - Reranked authority retrieval
- Priority: P1
- Area: Legal authority
- Problem: Authority search should rank source-backed ORS provisions by usefulness to matter facts/claims.
- Expected behavior: Rerank relevant provisions, definitions, deadlines, remedies, penalties, exceptions, and source notes, with currentness and scope warnings.
- Implementation notes: Use existing rerank service when configured and expose retrieval metadata, currentness state, source note IDs, and definition-scope caveats. Complaint authority/citation linking consumes this in `CB-CE-009`.
- Acceptance checks: Authority results show why they were recommended, what claim/element they support, whether they are current, and any scope/currentness warnings.
- Dependencies: V0 authority search, ORSGraph rerank config, `CB-CE-009`.
- Status: Todo

## CB-V02-010 - Strategy board
- Priority: P2
- Area: Strategy
- Problem: Strategy signals are not gathered into a work board.
- Expected behavior: Claim strength, defense strength, evidence completeness, authority strength, deadline risk, damages support, settlement leverage notes, weakness map, and next tasks.

## CB-V02-011 - Export package model and status API
- Priority: P1
- Area: Export
- Problem: Export needs durable package state rather than one-shot stub endpoints.
- Expected behavior: Add `ExportPackage` records with included WorkProduct IDs, exhibit IDs, generated artifact IDs, warnings, QC status, created timestamp, and download status.
- Implementation notes: Add routes for create package, get package status, regenerate package, and download generated DOCX/PDF artifacts. Shared WorkProduct export behavior is tracked in `CB-WPB-012`; complaint export artifacts and endpoint shape are tracked in `CB-CE-014`.
- Acceptance checks: User can create an export package preview, refresh status after generation, download available artifacts, and see blocking QC warnings.
- Dependencies: `CB-V02-005`, `CB-V02-006`, `CB-V02-007`, `CB-WPB-042`, `CB-CE-014`.
- Status: Partial
- Progress: Individual WorkProduct export artifacts, immutable snapshot links, hashes, warnings, and changed-since-export state exist. Multi-document export packages, status polling, regeneration, downloads, and filing packet assembly remain.

## CB-V02-012 - Exhibit numbering and packet ordering rules
- Priority: P1
- Area: Exhibits/export
- Problem: Exhibit references can become unreliable if labels shift during drafting or export.
- Expected behavior: Maintain stable exhibit labels, packet order, source document mapping, WorkProduct references, and regeneration warnings when labels conflict.
- Implementation notes: Use explicit exhibit records and avoid deriving labels only from document order. Complaint exhibit chips/references must warn on conflicts without silently changing existing references.
- Acceptance checks: Reordering a packet does not silently change existing draft exhibit references; conflicts surface as QC/export warnings.
- Dependencies: `CB-V02-004`, `CB-V02-005`, `CB-CE-008`, `CB-CE-018`.
- Status: Todo

## CB-V02-013 - Export visual verification workflow
- Priority: P2
- Area: Export quality
- Problem: DOCX/PDF output needs layout verification before users rely on filing packets.
- Expected behavior: Generated DOCX/PDF artifacts are rendered or inspected for headings, numbering, citations, exhibit references, page breaks, and overlap.
- Implementation notes: Keep generated status as review-needed until basic render checks pass. Complaint court-paper preview and export verification are tracked in `CB-CE-013` through `CB-CE-015`.
- Acceptance checks: Export tests fail on unreadable output, missing sections, broken numbering, or missing exhibit references.
- Dependencies: `CB-V02-006`, `CB-V02-007`, `CB-WPB-040`, `CB-WPB-041`, `CB-WPB-055`, `CB-CE-013`, `CB-CE-015`.
- Status: Todo

## CB-V02-014 - Authority recommendation attaches by context
- Priority: P1
- Area: Legal authority
- Problem: Current recommendation behavior is search-like and does not use selected claim/element/draft context deeply enough.
- Expected behavior: Recommendation accepts context IDs and returns ranked authorities with support target, reason, currentness, scope, and attach action metadata.
- Implementation notes: Use `claim_id` and later `element_id`, `draft_id`, `paragraph_id`, or `sentence_id`; never attach automatically without user review.
- Acceptance checks: Recommending authority for a claim or paragraph explains the target and allows one-click attach through the V0 authority attach endpoint.
- Dependencies: `CB-V0-025`, `CB-V02-008`.
- Status: Todo
