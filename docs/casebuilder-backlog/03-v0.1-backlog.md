# 03 - V0.1 Backlog

V0.1 expands the V0 case graph into answer/defense workflows, deadline/notice support, graph viewing, QC dashboards, task management, and the next WorkProduct profiles after complaint.

## CB-V01-001 - Answer profile
- Priority: P1
- Area: Answer workflow
- Problem: Users served with a complaint need paragraph-by-paragraph response support.
- Expected behavior: Upload complaint, extract allegations, build admit/deny/lack-knowledge grid, attach evidence, suggest defenses, generate `WorkProduct(type="answer")`, run QC, and preview/export through shared WorkProduct Builder.
- Implementation notes: Reuse CaseDocument, Fact, Evidence, Defense, WorkProduct, and WorkProductFinding models; add an answer response-grid DTO as profile-specific state instead of burying allegation responses in free-form draft text.
- Acceptance checks: User can generate an answer WorkProduct from allegation responses.
- Dependencies: V0 extraction, facts, evidence, defenses, WorkProduct Builder.
- Status: Todo

## CB-V01-002 - Complaint parser
- Priority: P1
- Area: Answer workflow
- Problem: Answer profile needs structured allegations from uploaded complaints.
- Expected behavior: Extract numbered paragraphs, parties, claims, prayer, exhibits, and response deadlines.
- Implementation notes: Start deterministic paragraph parsing for numbered pleadings; AI parser provider-gated.
- Acceptance checks: Parsed allegations retain page/paragraph source spans.
- Dependencies: `CB-V0-004`.
- Status: Todo

## CB-V01-003 - Defense Builder
- Priority: P1
- Area: Defenses
- Problem: Defense routes and backend endpoints exist, but the defense workflow is not complete.
- Expected behavior: Suggest procedural and affirmative defenses, map facts/evidence/authority, score risk, and draft answer paragraphs.
- Implementation notes: Use `CaseDefense` for standalone defenses and `Claim kind=defense` for unified element UI only if mapper requires it.
- Acceptance checks: Defense appears in evidence matrix, authorities, tasks, answer draft, and QC.
- Dependencies: V0 evidence, facts, authority search.
- Status: Partial

## CB-V01-004 - Deadline detection
- Priority: P1
- Area: Deadlines
- Problem: Deadline page exists, but live deadline extraction/detection is missing.
- Expected behavior: Detect deadlines from uploaded documents, ORSGraph deadlines, user-entered dates, and court notices.
- Implementation notes: Add DeadlineInstance creation endpoint if needed beyond read list; keep source law/document and trigger event mandatory or explicitly unknown.
- Acceptance checks: Deadline shows trigger event, source document/law, confidence, status, and tasks.
- Dependencies: V0 extraction and ORSGraph deadline retrieval.
- Status: Todo

## CB-V01-009 - QC dashboard
- Priority: P1
- Area: QC/risk
- Problem: QC page shows simple derived metrics only.
- Expected behavior: Unsupported allegations, uncited legal claims, bad citations, missing elements, weak evidence, contradictions, deadline risks, source conflicts, repeated facts, and unsupported WorkProduct blocks/sentences.
- Implementation notes: Store QC findings as graph nodes and link them to affected facts/elements/work products; use explicit QC run endpoints so findings are reproducible. WorkProduct-level QC and AST-targeted findings exist; matter-level QC aggregation remains. Complaint-specific QC categories and targets are tracked in `CB-CE-011`; shared lifecycle work is tracked in `CB-WPB-011`.
- Acceptance checks: Each QC item links to the exact source and affected work product.
- Dependencies: V0 fact/citation checks, evidence mapping, `CB-WPB-036`, and `CB-CE-011`.
- Status: Partial
- Progress: WorkProduct QC, AST validation warnings, and AST-targeted rule findings exist. The dashboard still needs matter-level aggregation, gap/contradiction nodes, and richer remediation flows.

## CB-V01-015 - QC run endpoint and finding lifecycle
- Priority: P1
- Area: QC/risk
- Problem: Findings can be generated by draft and WorkProduct actions, but there is no matter-level QC run or complete resolve/ignore/reopen lifecycle across all finding types.
- Expected behavior: Add `/api/v1/matters/:id/qc/run` producing a `QcRun` with findings, and endpoints to resolve, ignore, reopen, comment, and link remediation tasks.
- Implementation notes: Start deterministic over current facts, evidence, claims, WorkProducts, legacy drafts, citations, deadlines, and source spans. Complaint profile and shared WorkProduct Builder should reuse this lifecycle rather than creating parallel finding systems.
- Acceptance checks: Running QC creates reproducible findings, each finding has status history, and resolving/ignoring updates QC dashboard counts.
- Dependencies: `CB-V0-016`, `CB-V0-024`, `CB-V0-025`, `CB-V01-011`, `CB-V01-012`, `CB-WPB-036`, `CB-CE-011`.
- Status: Partial
- Progress: WorkProduct-level QC run and finding status patching exist, and findings synchronize into the AST/Case History. Matter-level `QcRun`, reopen/comment lifecycle, remediation task links, and cross-type aggregation remain.

## CB-V01-016 - Finding resolution endpoints
- Priority: P1
- Area: Validation
- Problem: Fact-check, citation-check, evidence-gap, authority-gap, and contradiction findings need consistent status transitions.
- Expected behavior: Provide matter-scoped resolve, ignore, reopen, and comment endpoints for validation findings.
- Implementation notes: Use one lifecycle shape across finding node types and log actor/action later through audit events. Complaint rule, support, citation, exhibit, formatting, and export findings should use the same lifecycle.
- Acceptance checks: All finding status changes are persisted, visible in editor/QC/graph, and cannot cross matter boundaries.
- Dependencies: `CB-V01-015`, `CB-WPB-036`, `CB-X-002`, `CB-CE-011`.
- Status: Partial
- Progress: WorkProduct findings can be patched and are mirrored into AST rule findings. Shared resolution routes for fact-check, citation-check, evidence-gap, authority-gap, contradiction, comments, and reopen remain.
